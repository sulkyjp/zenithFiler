## 問題の原因

1. タイミングの問題: `LoadDirectoryAsync` の `finally` ブロックで `LoadThumbnailsForCurrentItemsAsync` を呼んでいますが、`LoadIconsAsync` の完了を待たずに実行される可能性があります。
2. `LoadIconsAsync` の処理順序: `LoadIconsAsync` は `allItems` に対してサムネイルを読み込みますが、`Items` コレクション内のアイテムを更新する際に、`DispatcherPriority.Background` で非同期更新しているため、完了前に `LoadThumbnailsForCurrentItemsAsync` が実行される可能性があります。
3. `MergeItems` との競合: `MergeItems` で既存アイテムを再利用する際、サムネイルがクリアされる可能性があります。

## 対策案

### 案1: `LoadIconsAsync` 完了後に補完読み込みを実行（推奨）

`LoadDirectoryAsync` の `finally` ではなく、`LoadIconsAsync` の直後に補完読み込みを実行します。

メリット:
- `LoadIconsAsync` 完了後に確実に実行される
- タイミングの問題を回避できる

### 案2: `LoadIconsAsync` 内でアイコンビュー時のサムネイル読み込みを強化

`LoadIconsAsync` 内で、現在のビューモードがアイコンビューの場合、サムネイル読み込みを確実に行うようにします。

メリット:
- 1箇所で処理を完結できる
- 補完処理が不要になる可能性がある

### 案3: `MergeItems` でサムネイルを保持

`MergeItems` で既存アイテムを再利用する際、サムネイルを保持するようにします。

メリット:
- 既存のサムネイルが失われない

デメリット:
- フォルダ移動時に古いサムネイルが残る可能性がある

## 推奨実装

案1と案2を組み合わせます。

1. `LoadIconsAsync` 完了後に補完読み込みを実行（案1）
2. `LoadIconsAsync` 内で、アイコンビュー時はサムネイル読み込みを確実に行う（案2）

## 調査結果

### 1. サムネイルが表示されない

スクリーンショットは**詳細ビュー**（名前・更新日時・種類・サイズ）です。サムネイルは**アイコンビュー**（大・中・小アイコン）でのみ使われます。

考えられる要因：
- **ビューモード**: 詳細ビューでは `ThumbnailOrIconConverter` は使われず、サムネイルは表示されません
- **タイミング**: `LoadIconsAsync` と `LoadThumbnailsForCurrentItemsAsync` の実行順や、`Items` の更新タイミング
- **外部ドライブ (J:)**: 外部SSDでは Shell API の応答が遅く、サムネイル取得が失敗しやすい可能性

### 2. 更新時の異常終了

想定される原因：

| 原因 | 説明 |
|------|------|
| **並行実行** | `Refresh()` が `_ = LoadDirectoryAsync()` で呼ばれ、前回の `LoadDirectoryAsync` が完了する前に新しい処理が始まる |
| **競合状態** | 古い `LoadIconsAsync` の Dispatcher コールバックが、新しい `MergeItems` で変更された `Items` を参照・更新する |
| **例外の握りつぶし** | `catch { }` で例外を無視しており、原因の特定が困難 |
| **Shell/COM** | `ShellFile.FromFilePath` や Shell API が外部ドライブで例外や AccessViolation を起こす可能性 |

## 改善案

### サムネイル表示の改善

1. **LoadDirectoryAsync の完了待ち**
   - `Refresh()` を `await LoadDirectoryAsync()` に変更するか、`Refresh` 実行中フラグを導入して二重実行を防ぐ

2. **アイコンビュー時のサムネイル優先**
   - アイコンビューでは、アイコン取得より先にサムネイルを読み込む処理を検討

3. **外部ドライブの考慮**
   - `ShellThumbnailService.GetThumbnail` で、外部ドライブやネットワークパス向けにタイムアウトやリトライを検討

### 異常終了の防止

1. **LoadDirectoryAsync の直列化**
   ```csharp
   // 例: セマフォで同時実行を1つに制限
   private readonly SemaphoreSlim _loadDirectorySemaphore = new(1, 1);
   ```
   - 新しい `LoadDirectoryAsync` 開始時に、前回の完了を待つ

2. **例外のログ出力**
   ```csharp
   catch (Exception ex)
   {
       _ = App.FileLogger.LogAsync($"LoadDirectoryAsync: {ex}");
   }
   ```
   - `catch { }` をやめ、最低限ログを残す

3. **Refresh の二重実行防止**
   - `Refresh()` 実行中フラグを設け、実行中は `Refresh()` を無視するか、キューイングする

4. **ShellThumbnailService の防御的処理**
   - `GetThumbnailOnStaThread` 内で try-catch を強化し、例外をログして null を返す
   - 必要なら、外部ドライブ用に別の取得ロジックやフォールバックを検討

5. **OnRefreshCompleted の安全性**
   - `list.Items` の列挙中に `Items` が変更されないよう、`ToList()` でコピーしてから処理する
   - `list` や `DataContext` の null チェックを確実に行う

### 実装の優先度

| 優先度 | 対策 | 効果 |
|--------|------|------|
| 高 | LoadDirectoryAsync の直列化 | 競合によるクラッシュの防止 |
| 高 | 例外のログ出力 | 原因の特定とデバッグ |
| 中 | Refresh の二重実行防止 | 不要な負荷と競合の軽減 |
| 中 | OnRefreshCompleted の防御的コーディング | コレクション変更時の例外防止 |
| 低 | ShellThumbnailService の強化 | Shell/COM 起因のクラッシュ軽減 |

## 3. 推奨する修正方針

| 箇所 | 対策 |
|------|------|
| `OnRefreshStarting` | `list.SelectedItems.Cast<FileItem>().Select(x => x.FullPath).ToList()` の前に、`list.SelectedItems` を一度 `ToList()` などでスナップショット化する。 |

## 結論：WPF レンダリングスレッドでのネイティブクラッシュ

ログから分かる事実を整理します：

1. **Refresh 処理は全ステップが正常に完了**している（175行「OnRefreshCompleted コールバック完了」まで出力）
2. **`DispatcherUnhandledException` のログが出ていない**
3. **`UnhandledException` のログも出ていない**（`LogSync` で同期書き込みしているにもかかわらず）

これが意味すること：**マネージド例外ではなく、プロセスが即死するネイティブ例外（Corrupted State Exception）が発生している**ということです。

### 発生メカニズム

1. `LoadIconsAsync` / `LoadThumbnailsForCurrentItemsAsync` でサムネイルとアイコンを取得し、`FileItem.Thumbnail` / `FileItem.Icon` にセット
2. `OnRefreshCompleted` でコレクション更新・選択復元が完了
3. その後の **WPF レンダリングフレーム** で、アイコンビューの `Image` 要素がバインドされた `BitmapSource` を描画しようとする
4. この描画は **WPF のレンダリングスレッド**（UI スレッドとは別のネイティブスレッド）で行われる
5. ここで不正な `BitmapSource`（破損したサムネイルデータ等）があると、**ネイティブレベルの AccessViolation が発生**してプロセスが即死

.NET では `AccessViolationException` 等の Corrupted State Exception は、デフォルトでは `try-catch` でも `UnhandledException` でも捕捉できません。

### 証拠

- パスが `J:\010_写真\2025\20250913-14\04_resize` — **外部ドライブの画像フォルダ**で、44個のファイルがある
- `ShellThumbnailService.GetThumbnailOnStaThread` で `ShellFile.FromFilePath` → `thumb.BitmapSource` を取得して `Freeze()` しているが、WindowsAPICodePack が返す `BitmapSource` が内部的に破損データを持っている場合、`Freeze()` は成功するが描画時にクラッシュする

### 推奨する修正方針

Agent モードに切り替えていただければ、以下を実装できます：

1. **`ThumbnailOrIconConverter` で防御的チェック** — `BitmapSource` の `PixelWidth` / `PixelHeight` が 0 以下なら null を返す
2. **`Image.ImageFailed` イベントのハンドル** — XAML の各 Image 要素に `ImageFailed` イベントを追加し、描画失敗時に Source を null にクリア
3. **サムネイル取得時の妥当性検証** — `GetThumbnailOnStaThread` で `BitmapSource` を取得後、`PixelWidth > 0` かつ `PixelHeight > 0` であることを確認してから返す
4. **`WriteableBitmap` への変換** — STA スレッドで取得した `BitmapSource` を一旦 `WriteableBitmap` にコピーしてから `Freeze()` することで、元の COM オブジェクトとの依存を切り、描画時のネイティブクラッシュを防止する

特に **4番の `WriteableBitmap` への変換**が最も効果的です。`ShellFile.FromFilePath` が返すサムネイルは内部的に COM オブジェクト（Shell の `IShellItemImageFactory`）に依存しており、`using var shellFile` でスコープを抜けると COM が解放されますが、`BitmapSource` がまだ内部参照を持っている可能性があります。`WriteableBitmap` にコピーすれば、完全に独立したピクセルデータになるためこの問題が解消されます。

## 根本原因の特定

4 ファイルでもクラッシュし、テンプレート内容・ControlTemplate・ItemsSource 共有が全て無関係ということは、**ScrollViewer と WrapPanel 系パネルの間のレイアウト測定ループ（measurement oscillation）** が原因です。

これは WPF の既知の問題パターンです:

1. `ScrollViewer.MeasureOverride` が WrapPanel/VirtualizingWrapPanel を測定
2. パネルが DesiredSize を報告 → ScrollViewer がスクロールバーの要否を判断
3. スクロールバーの表示/非表示で**利用可能な幅が変わる**
4. パネルを新しい幅で**再測定** → 折り返し位置が変わる → DesiredSize が変わる
5. → 手順 2 に戻る（無限ループ → StackOverflow）

`VirtualizingStackPanel`（詳細ビュー）ではこの問題が起きません。各アイテムが 1 行を占めるため、スクロールバーの有無で行数が変わらないからです。

## 提案する方針 H

**ScrollViewer の垂直スクロールバーを常時表示（`Visible`）** にすることで、幅の変動を排除します。

```xml
ScrollViewer.VerticalScrollBarVisibility="Visible"
```

スクロールバーが常に表示されていれば、利用可能な幅は常に一定になり、測定ループが発生しません。

これでも解決しない場合は、`WrapPanel` / `VirtualizingWrapPanel` を **`UniformGrid`**（固定列数）に置き換えます。`UniformGrid` は幅依存の折り返しを行わないため、この問題が原理的に発生しません。

## コードから推測できる再帰の可能性

メソッド名がまだ分からなくても、**H-2 で追加した処理**で次のようなループが考えられます。

- `FileListBox_SizeChanged`  
  → `UpdateIconColumnCount(ActualWidth)`  
  → `IconColumnCount` を更新（DependencyProperty）  
  → UniformGrid の `Columns` が変わりレイアウトが走る  
  → 内部でサイズが変わり、再度 `FileListBox_SizeChanged` が発生  
  → また `UpdateIconColumnCount` … の **再入（再帰）**

この場合、クラッシュは `TabContentControl` の `FileListBox_SizeChanged` や `UpdateIconColumnCount` 周辺のメソッド（トークン 660 付近）で起きている可能性が高いです。

**対策案（実装するときは Agent モードで）:**

1. **再入防止**  
   `FileListBox_SizeChanged` の先頭でフラグを立て、`UpdateIconColumnCount` を実行する間は `true`、終了時に `false` に戻す。フラグが `true` のときは `UpdateIconColumnCount` を呼ばない。
2. **値が変わるときだけ更新**  
   `UpdateIconColumnCount` 内で「計算した列数」と「現在の `IconColumnCount`」を比較し、**違うときだけ** `IconColumnCount` をセットする（同じ値での再レイアウトを防ぐ）。
3. **非同期で1回だけ反映**  
   `Dispatcher.BeginInvoke(DispatcherPriority.Loaded, ...)` で「列数更新」を 1 回だけキューに載せ、同期的な SizeChanged の連鎖を切る。

まずは **2（値が変わるときだけ更新）＋ 1（再入防止）** を入れると安全です。

- メソッド特定は **`bin\Debug\net8.0-windows` で `LoadFrom`** か **ILSpy で 0x06000660 を検索**するとよいです。
- それと並行して、**SizeChanged → UpdateIconColumnCount → レイアウト → SizeChanged のループを防ぐ修正**を Agent モードで入れる価値は高いです。

Agent モードに切り替えてもらえれば、再入防止と「値が変わるときだけ更新」を具体的なコード変更案として書きます。

## 変更候補

**1. F5（更新）をアイコンビューでは実行しない**

`TabContentControl.xaml.cs` の `ListView_KeyDown` 内、F5 の処理で「詳細ビューのときだけ実行」にします。

```csharp
// F5 (Refresh)
else if (e.Key == Key.F5)
{
    if (vm.FileViewMode != FileViewMode.Details) { e.Handled = true; return; } // アイコンビューでは更新しない
    App.FileLogger.LogSync("[F5] KeyDown F5 受信");
    vm.RefreshCommand.Execute(null);
    ...
}
```

**2. キーボード操作をアイコンビューでは無効にする**

同じ `ListView_KeyDown` の**先頭**で、送信元が `FileListBox`（アイコンビュー）のときは、F5 以外のキーも含めて「何もせず Handled のみ」にすると安全です。

例: ハンドラの冒頭で

```csharp
if (sender is ListBox listBox && listBox == FileListBox)
{
    e.Handled = true;  // 矢印・Enter・Back・Delete 等を無効化
    return;
}
```

こうすると、アイコンビューでは一覧のキーボード操作は一切効かず、クリックでの選択・ダブルクリックでの開くだけになります。  
「矢印だけ無効にして Enter は効かせる」など細かくしたい場合は、キーごとに `if (e.Key == Key.F5) ... else if (e.Key == Key.Enter) ...` と分岐する形にできます。

**3. 更新ボタン（ツールバー等）の無効化**

画面上に「更新」ボタンがある場合は、`FileViewMode != FileViewMode.Details` のときは `Command` を無効（`CanExecute = false`）にするか、実行時に「詳細ビューでないなら何もしない」分岐を入れる必要があります。該当ボタンのバインドやコマンド実装を確認する必要があります。

---

まとめると、

- **F5** … アイコンビュー時は `RefreshCommand` を実行しない（上記 1）。
- **キーボード** … アイコンビュー時は `ListView_KeyDown` で即 `e.Handled = true; return;`（上記 2）。

この方針でコードを書き換えたい場合は、Agent モードに切り替えてもらえれば、該当ファイルを開いてそのまま修正案を適用できます。
